# AgentHub 项目背景、现状理解与实施计划

更新时间：2026-05-26

## 1. 项目定位

AgentHub 不是单一工具，而是一个面向行业能力的 AI 能力市场与运行平台。

当前平台的核心方向是：

- 对客户：订阅可落地的行业 AI 能力，例如 ISP 智能客服、AI 智能监控。
- 对开发者：后续开放能力训练、能力打包、上传、审核、发布和收益分成。
- 对平台：统一管理账号、订阅、配置、能力调用、边缘客户端、通知、计费、审核和能力市场展示。

现阶段最重要的产品原则是：`首页少说，详情页说透；配置要能落地；能力必须可订阅、可接入、可验证、可持续训练。`

## 2. 当前已经形成的能力

### 2.1 AI 能力市场

首页已经调整为简约能力市场，重点展示能力模块、搜索、分类和开发者入驻入口。这个方向是正确的，因为后续平台会承载更多开发者上传的能力，不适合做成传统官网式页面。

### 2.2 ISP 智能客服

ISP 智能客服的定位是整体打包能力，价格已经调整为：

- 499 元/月
- 4990 元/年

当前理解：

- 客户不一定使用同一套计费系统、OLT 管理系统、TR069 光猫管理系统、IPTV 系统。
- 配置页应该按系统类型选择适配器，而不是默认写死凌风或秒开。
- 当前已训练或优先支持的系统包括：
  - 凌风计费系统
  - 秒开 OLT 管理系统
  - 秒开 TR069 管理系统
  - 智慧光迅 IPTV 系统
- 后续要逐步增加常见 ISP 系统适配器。
- 通知不应按客服、装维、老板这些角色拆死，而应该默认作为能力通用能力存在。客服同样可能查询宽带故障、用户在线状态、光猫状态、续费结果和工单异常。

因此 ISP 能力配置应包含：

- 计费系统适配器
- OLT 管理系统适配器
- TR069 光猫管理系统适配器
- IPTV 系统适配器
- 多通道业务通知：微信/企微、飞书、邮件、短信、自定义 Webhook
- 业务通知事件：账号查询、故障诊断、在线状态、光猫状态、续费结果、工单创建、系统异常、日报汇总

### 2.3 AI 智能监控

AI 智能监控的定位是面向物业、小区、园区、商业场所的打包能力，当前定价为：

- 990 元/月
- 9900 元/年

当前理解：

- 客户现场通常已经有 NVR 录像主机，历史录像来源默认来自 NVR。
- 摄像头实时接入主要依赖 RTSP / ONVIF。
- NVR 的录像查询方式按品牌和型号差异较大，需要做品牌适配器，而不是把“录像来源”作为客户配置分类。
- AI 找人不是普通录像回查的同义词，它属于录像回查之上的专项能力，应作为能力类型展示和训练。
- 卫生巡检、高空抛物、物品遗失、AI 找人、录像回查等能力对应的通知对象和处理流程不同，必须支持按事件类型配置不同通知通道和联系人。

AI 监控能力类型建议保持以下分类：

- 实时告警：卫生巡检、高空抛物、人员跌倒、区域入侵等。
- 录像回查：按摄像头、位置、时间段检索 NVR 历史录像。
- AI 找人：基于衣着、体貌、出现时间、摄像头路径做跨片段检索。
- 物品遗失/拿取分析：定位物品出现、消失、被拿走的时间线。
- 事件复盘：生成事件摘要、关键帧、时间线和责任通知。

## 3. 当前架构理解

### 3.1 前端

技术栈：Vue + Vite + Tailwind。

主要页面：

- 能力市场首页
- 能力详情页
- 注册/登录
- 控制台
- 开发者中心
- 管理后台
- 独立配置页 `cap-config.html`
- 独立能力对话页 `cap-chat.html`

当前不合理点：

- SPA 页面和独立 HTML 配置页并存，短期可用，但长期应统一到 Vue 路由中。
- 控制台、能力详情、配置页之间的订阅状态、能力状态、API Key 和客户端下载入口需要形成一致闭环。

### 3.2 后端

技术栈：FastAPI + SQLite + SQLAlchemy，部分视觉接口直接使用 sqlite3。

主要模块：

- 用户认证
- 市场能力
- 订阅和 API Key
- 能力配置
- 能力对话
- 开发者能力管理
- 管理员审核
- AI 视觉设备、摄像头、事件、通知、规则
- WebSocket 客户端桥接

当前不合理点：

- AI 视觉接口中仍存在 `demo_user` 占位逻辑，必须改为真实登录用户或设备归属。
- 视觉设备、通知、规则、事件列表缺少完整租户隔离。
- 视觉事件已经保存视频片段文件，但事件入库字段还需要补齐 `video_clip_path`。
- AI 监控对话当前更多是工具框架，真实 NVR 录像检索、片段分析、AI 找人还需要适配器和训练数据支撑。
- 开发者上传能力已有接口雏形，但缺少能力包规范、自动检测、沙箱验证、协议规则和发布审核标准。

### 3.3 客户端与边缘端

当前存在两个概念：

- `client/`：统一 WebSocket 客户端，已支持 ISP 插件。
- `edge/`：AI 监控边缘端，负责 ONVIF/RTSP、模型推理、事件上报、通知同步。

当前不合理点：

- 两套客户端没有完全统一，下载入口也没有形成正式安装包版本体系。
- `client/main.py` 尝试加载 `plugins.vision`，但当前视觉能力主要在 `edge/` 中实现。
- 后续应该收口为 `AgentHub 统一客户端`，内部按能力插件加载：
  - ISP 插件
  - AI 监控插件
  - NVR 品牌适配器插件
  - 通知/本地代理插件

## 4. 需要优先修正的不合理点

### P0：必须先收口

1. 视觉接口租户隔离  
   去掉 `demo_user`，设备注册、通知渠道、规则、事件必须绑定真实用户或订阅。

2. 统一客户端下载与版本  
   平台应提供明确的 Windows 安装包、Linux 安装脚本、版本号、更新说明和校验值。

3. 订阅到使用闭环  
   用户订阅后应能直接看到：配置能力、下载客户端、复制 API Key、打开能力对话、查看事件或调用日志。

4. NVR 适配器契约  
   先定义统一接口，再做品牌适配，不要在对话逻辑里写死品牌。

5. 敏感信息治理  
   前端不能显示 AI 模型、内部 Key、供应商密钥、服务器细节。所有密钥只放后端环境变量或加密配置中。

6. 视觉事件存储补齐  
   事件表已经有 `video_clip_path` 字段，上报视频片段时必须入库，方便后续录像回查和事件复盘。

### P1：产品能力完善

1. ISP 多系统适配器从配置项升级为真实适配器注册表。
2. AI 监控通知规则支持按事件、摄像头、时间段、严重级别、负责人路由。
3. 开发者入驻介绍页、入驻协议、能力发布规则、收益规则上线。
4. 能力包 manifest 规范、上传校验、自动扫描和沙箱测试上线。
5. 控制台增加能力运行状态、客户端在线状态、通知测试和错误提示。

### P2：规模化运营

1. 能力评分、调用统计、收入统计、退款和发票。
2. 开发者能力质量分和违规处罚机制。
3. 模型训练数据闭环：样本收集、标注、评测、发布、回滚。
4. 多租户审计日志、管理员操作记录、客户数据导出。

## 5. AI 监控训练与落地计划

AI 监控不是一次训练完所有能力，而是按“可验证场景”逐个沉淀。

### 第 0 阶段：现场资料与标准准备

时间：第 1 周同步启动。

你需要准备：

- 客户类型：物业、小区、园区、学校、工厂等。
- NVR 品牌、型号、固件版本、账号权限、内网访问方式。
- 摄像头点位表：位置、通道号、视角、是否夜视、是否有遮挡。
- 典型事件定义：什么算卫生不合格、什么算高空抛物、什么算物品遗失。
- 通知对象和处置 SOP：例如卫生巡检通知清洁主管，高空抛物通知保安队长。
- 脱敏样本视频：每类事件至少准备正样本、负样本、模糊样本。
- 验收标准：误报率、漏报率、响应时间、能否生成可用结论。

我负责：

- 整理训练样本目录结构。
- 制定标注规范。
- 设计能力包 manifest。
- 把训练结果、适配器、配置模板、测试用例打包进平台。

### 第 1 阶段：NVR 录像回查底座

时间：第 2-4 周。

目标：

- 不先训练复杂视觉模型，先打通“能查录像、能拿片段、能给 AI 分析”的底座。
- 优先做一个最常见品牌的 NVR 适配器，建议从大华或海康开始。

训练/开发方式：

- 定义 NVR 统一接口：
  - 登录
  - 获取通道列表
  - 按通道和时间查询录像
  - 拉取片段或生成播放地址
  - 获取关键帧
- 对自然语言做结构化解析：
  - 地点
  - 摄像头
  - 时间范围
  - 目标对象
  - 任务类型
- 形成测试问题集，例如：
  - “查一下大厅下午 3 点左右桌上手机是谁拿走的”
  - “看下 2 栋门口昨天晚上有没有人搬东西”
  - “回放一下车库入口 10 分钟前的画面”

你需要准备：

- NVR 测试环境或远程可访问环境。
- NVR 品牌接口文档或可测试账号。
- 典型通道和录像片段。
- 允许测试的时间段。

### 第 2 阶段：卫生巡检能力

时间：第 4-6 周。

目标：

- 识别垃圾堆放、地面明显脏污、垃圾桶满溢、公共区域异常杂物。
- 支持定时巡检和告警通知。

训练方式：

- 先使用目标检测/图像分类模型做场景识别。
- 结合规则阈值减少误报，例如连续多帧确认、指定区域检测、巡检时间段。
- 按客户现场不断补充正负样本。

你需要准备：

- 干净/不干净对比视频或图片。
- 垃圾、污渍、堆物、垃圾桶满溢样本。
- 不应报警的负样本：阴影、反光、临时路过人员、正常摆放物。
- 每个场景的处置人和通知方式。

### 第 3 阶段：高空抛物能力

时间：第 6-8 周。

目标：

- 识别从高处向下运动的异常物体。
- 生成事件时间、疑似楼层/区域、关键帧和告警通知。

训练方式：

- 使用运动轨迹检测 + 物体检测 + 多帧确认。
- 真实样本不足时，先用安全可控的模拟样本训练和评测。
- 夜间、雨天、反光、树叶飘落要作为负样本重点处理。

你需要准备：

- 可合法使用的模拟视频。
- 楼栋摄像头安装位置、朝向、覆盖楼层。
- 高空抛物事件处置流程。
- 保安队长或物业负责人通知通道。

### 第 4 阶段：AI 找人与物品遗失

时间：第 8-12 周。

目标：

- AI 找人：按衣着、外观描述、出现时间和摄像头路径找人。
- 物品遗失：判断物品出现、消失、被拿走的大致时间线。

训练方式：

- AI 找人不建议一开始做强人脸识别，应优先做行人重识别、衣着特征、轨迹检索。
- 物品遗失使用目标检测、区域变化、关键帧对比和时间线生成。
- 输出要保持谨慎：提供“疑似片段”和“可信度”，不要直接下定论。

你需要准备：

- 有授权的人员样本和监控片段。
- 摄像头点位拓扑。
- 典型物品类别：手机、包、快递、工具等。
- 法务或客户授权边界，避免隐私风险。

## 6. ISP 能力训练与扩展计划

### 第 1 阶段：已有系统固化

时间：第 1-2 周。

目标：

- 凌风计费、秒开 OLT、秒开 TR069、智慧光迅 IPTV 形成稳定适配器。
- 每个适配器都有配置字段、连接测试、常用操作和失败提示。

你需要准备：

- 测试账号。
- 常见客服问题清单。
- 标准操作流程。
- 异常情况样本：用户不存在、系统超时、设备离线、权限不足。

### 第 2 阶段：新增系统适配器

时间：第 3 周开始滚动迭代。

方式：

- 每新增一个系统，先收集接口或 UI 操作流程。
- 建立“动作契约”，例如查询用户、查询在线状态、查询光猫、查询 OLT 端口、创建工单。
- 训练自然语言到工具调用的样本。
- 做回归测试，确保不同系统同类动作返回统一结构。

## 7. 开发者生态实施计划

### 第 1 阶段：入驻介绍和规则

时间：第 1 周。

要上线的内容：

- 开发者入驻介绍页。
- 平台支持的能力类型。
- 能力发布流程。
- 收益分成说明。
- 禁止能力说明。
- 数据合规和隐私要求。
- 入驻协议和能力发布协议。

协议至少包含：

- 知识产权归属。
- 数据来源合法性。
- 客户隐私保护。
- 禁止上传恶意代码。
- 能力质量和 SLA 要求。
- 收益结算和退款责任。
- 平台下架、冻结和处罚权利。

### 第 2 阶段：能力包规范

时间：第 2 周。

能力包必须包含：

- `manifest.json`：能力 ID、名称、版本、分类、运行方式、价格、权限声明。
- `README.md`：能力说明、适用场景、限制。
- `config_schema.json`：客户配置字段。
- `tool_schema.json`：可调用工具和参数。
- `tests/`：测试用例。
- `examples/`：示例输入输出。
- `security.md`：数据、网络、密钥和权限声明。
- 代码或适配器文件。

### 第 3 阶段：上传与审核流程

时间：第 3-4 周。

流程：

1. 开发者注册并完成资料。
2. 创建能力草稿。
3. 上传能力包。
4. 平台自动校验：
   - manifest 是否完整
   - 版本号是否合法
   - 是否含敏感信息
   - 是否有危险系统调用
   - 是否包含测试用例
5. 沙箱运行测试。
6. 人工审核：
   - 功能是否真实
   - 说明是否准确
   - 价格是否合理
   - 是否侵犯隐私或违反法律
   - 是否有客户可理解的配置说明
7. 进入灰度发布。
8. 正式发布到市场。
9. 运行监控和投诉处理。

### 第 4 阶段：开发者开放节奏

建议在两个官方能力稳定后开放：

- ISP 智能客服完成至少 2-3 套系统适配器。
- AI 智能监控完成 NVR 回查底座和至少 1 个专项能力。
- 订阅、配置、客户端、通知、对话形成闭环。
- 审核流程、协议、能力包规范已经上线。

## 8. 分阶段实施路线

### 第 1 周：平台闭环和文档规则

- 修正视觉接口租户隔离设计。
- 补齐统一客户端下载说明和版本策略。
- 上线开发者入驻介绍、协议、发布规则草案。
- 固化 ISP 和 AI 监控配置模板。
- 建立 AI 监控样本收集规范。

### 第 2 周：订阅、配置、客户端闭环

- 用户订阅后能完整走通：详情页 -> 配置 -> 下载客户端 -> 复制 API Key -> 测试连接 -> 打开对话。
- ISP 适配器配置做连接测试。
- AI 监控通知渠道做测试发送。
- 视觉事件补齐视频片段路径。

### 第 3-4 周：NVR 与开发者审核

- 定义 NVR 适配器接口。
- 完成第一个 NVR 品牌适配器。
- 开发者能力包 manifest 校验。
- 上传包自动扫描和沙箱测试雏形。
- 管理后台增加审核 checklist。

### 第 2 个月：AI 监控专项能力训练

- 卫生巡检进入可用 Beta。
- 高空抛物进入可用 Beta。
- 通知规则按事件负责人路由。
- 形成第一批客户现场评测报告。

### 第 3 个月：AI 找人、物品遗失和生态内测

- AI 找人和物品遗失进入内测。
- 开发者入驻开放邀请制。
- 能力上传、审核、灰度发布跑通。
- 建立能力质量分、投诉和下架机制。

## 9. 你需要配合准备的材料清单

### AI 监控

- 2-3 个真实或模拟客户场景。
- NVR 品牌、型号、固件版本、通道列表。
- 可测试账号或远程测试环境。
- 摄像头点位表和截图。
- 每类能力至少 20-50 段样本片段，先少量也可以启动。
- 通知对象：姓名、职责、微信/飞书/邮箱/短信方式。
- 每类事件的处理 SOP。
- 数据授权和隐私边界确认。

### ISP 智能客服

- 各系统测试账号。
- 常见客服话术和问题清单。
- 常见操作流程截图或录屏。
- 异常情况样本。
- 客户希望开放给客服的查询权限边界。

## 10. 下一步建议

我建议下一步先做 P0 收口，不急着训练所有监控能力：

1. 修正 AI 视觉接口的 `demo_user` 和租户隔离。
2. 补齐视觉事件 `video_clip_path` 入库。
3. 把统一客户端下载、API Key、配置测试做成订阅后的清晰闭环。
4. 新建开发者入驻介绍、协议、能力包规范页面。
5. 定义 NVR 适配器接口，然后选择一个品牌做第一个真实适配器。

这样平台会先从“能展示”变成“能交付”，后面的训练和开发者生态才有可靠地基。
