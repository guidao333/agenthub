# AgentHub 客户端与模型体系开发需求

## 1. 背景

AgentHub 的能力不应只依赖云端页面。客户安装客户端后，客户端应成为客户内网里的本地代理、任务执行器、资料处理入口和安全边界。

对于 ISP 智能客服，除了计费系统、OLT、TR069、IPTV、工单系统适配外，还需要支持客户本地运营工作，例如工资统计、广告商佣金核算、收入报表、客服话术整理、日报月报生成等。

这些能力分为两类：

- AgentHub 官方能力：由平台训练、封装、授权和计费，属于平台核心能力成果。
- 客户私有能力：由客户基于自身规则、表格、制度和资料训练或配置，只在客户自己的账号和客户端下生效。

## 2. 核心原则

1. 官方能力成果由平台掌控。
2. 客户私有规则允许本地记忆和本地训练，但不自动进入公共能力市场。
3. 客户自备模型可以支持，但优先通过 AgentHub 模型网关接入。
4. 客户端可以有本地对话窗口，但任务编排、授权、审计和能力边界仍由平台控制。
5. 客户端只开放出站连接到 AgentHub，不在客户内网暴露公网入口。
6. 插件动作必须白名单化，不能让云端或模型任意执行本地命令。

## 3. 客户端功能范围

### 3.1 基础客户端能力

- API Key 绑定、解绑、换绑。
- 设备 ID 生成与持久化。
- 客户身份、订阅能力、能力权限校验。
- WebSocket 长连接。
- 心跳上报：在线状态、版本号、设备 ID、系统信息。
- 插件加载状态上报：能力 ID、插件版本、支持动作列表。
- 配置同步：从平台拉取能力配置，支持热更新。
- 任务分发：按 `capability_id + action` 调用插件。
- 任务回传：结构化结果、错误、耗时、日志摘要。
- 超时、失败、重试、取消任务。
- 本地日志与诊断包。
- 敏感字段脱敏：API Key、密码、Webhook、模型 Key 不写入明文日志。
- Windows 开机自启或后台服务。
- Linux systemd 支持。
- 客户端版本检测与升级提示。

### 3.2 ISP 智能客服插件能力

- 计费系统适配。
- OLT 管理系统适配。
- TR069/ACS 光猫管理系统适配。
- IPTV 系统适配。
- 工单系统适配。
- 宽带账号查询。
- 在线状态查询。
- 光猫/ONU 状态查询。
- 故障诊断。
- 续费、停机、复机。
- 派发工单。
- 内网连通性检测。
- 客服文字工作：话术生成、投诉整理、回访记录。
- 运营报表：收入日报、套餐统计、欠费统计、续费统计。
- 财务结算：工资统计、广告商佣金核算、渠道分润。

### 3.3 AI 监控边缘能力

- 摄像头 RTSP/ONVIF 接入。
- NVR 录像主机适配。
- 实时告警。
- 卫生巡检。
- 高空抛物。
- 录像回查。
- AI 找人。
- 本地截图、片段保存、事件上报。
- 摄像头状态、推理 FPS、CPU/GPU/内存上报。
- 按事件路由通知：卫生巡检通知清洁主管，高空抛物通知保安队长等。

## 4. 模型提供方式

### 4.1 默认模式：平台提供模型

普通 SaaS 客户默认使用平台模型。

适用：

- ISP 智能客服。
- AI 监控告警分析。
- 客服话术。
- 故障诊断。
- 报表解释。
- 普通客户私有工作流。

要求：

- 平台统一做模型调用、限流、计费、审计。
- 官方能力提示词、工具编排、Function Calling 定义保留在平台侧。
- 客户只看到配置、结果和必要的执行日志。

### 4.2 高级模式：客户自备云模型

客户可提供自己的模型 API Key 或 OpenAI-compatible API。

要求：

- 通过 AgentHub 模型网关接入。
- 平台仍负责能力编排、工具权限、审计和质量评测。
- 不直接把完整官方能力逻辑下发给客户。
- 每个能力可配置允许使用的模型范围。
- 客户模型必须通过基础能力评测后才能用于生产。

### 4.3 私有化模式：客户本地模型

适用于政企、园区、学校、医院等对数据出域敏感的客户。

要求：

- 仅对企业版/私有化部署开放。
- 官方能力包需要签名和授权。
- 本地模型能力效果需要验收。
- 平台可降级为授权、审计和升级中心。

## 5. 能力成果保护

官方能力成果包括：

- 行业提示词模板。
- 工具调用协议。
- Function Calling schema。
- 能力编排流程。
- 系统适配器。
- 诊断逻辑。
- 安全校验。
- 审核规则。
- 评测用例。
- 回退策略。

保护要求：

- 官方能力包签名校验。
- 插件动作白名单。
- 服务端保留核心编排逻辑。
- 客户端不保存完整官方提示词。
- 客户自备模型调用优先走模型网关。
- 敏感配置加密保存或最小化保存。
- 能力调用全链路审计。
- 禁止客户私有训练数据自动进入公共能力市场。

## 6. 第三方本地智能体旁路风险

客户电脑上可能安装 OpenClaw、Codex、AutoGPT、远控软件、脚本工具或其它本地智能体。这些工具理论上可以读取本机文件、监控进程、调用本地接口、打包目录或上传文件，因此必须承认：

只要官方能力成果以完整明文形式落到客户机器上，就无法绝对防止被其它智能体或人工导出。

因此 AgentHub 客户端设计必须遵循“平台掌握大脑，客户端掌握手脚”的原则。

### 6.1 风险场景

- 读取客户端安装目录。
- 打包插件源码。
- 查看本地提示词、规则库、能力配置。
- 截获日志和运行参数。
- 调用本地 Web UI 或本地 API。
- 逆向能力包。
- 拷贝模型权重或适配器。
- 尝试把官方能力迁移到其它平台。

### 6.2 防护原则

1. 官方能力核心逻辑不完整下发。
2. 客户端只保留必要连接器、协议适配器和本地执行器。
3. 官方提示词、工具编排、Function Calling 策略、行业知识链、评测规则、授权判断、计费逻辑优先保留在平台侧。
4. 客户端不保存完整官方提示词。
5. 客户端不保存未订阅能力包。
6. 客户端不提供导出官方能力包、读取插件目录、压缩安装目录、执行任意命令等接口。
7. 本地 Web UI 只能作为工作台，不作为能力包管理后门。
8. 官方能力包必须签名、授权、可审计。
9. 客户私有能力和官方能力必须目录隔离、权限隔离、导出策略隔离。

### 6.3 目录隔离建议

客户端本地目录建议分为：

- `official_capabilities/`：官方能力包，加密、签名、绑定订阅，禁止导出。
- `private_workspace/`：客户私有规则、报表模板、上传资料，可备份、可导出、可删除。
- `logs/`：运行日志和任务日志，必须脱敏。
- `config/`：客户端配置，敏感字段加密或最小化保存。
- `cache/`：临时缓存，可清理，不保存长期敏感能力成果。

### 6.4 本地接口约束

如果客户端提供 `127.0.0.1` 本地 Web UI 或本地 API，必须满足：

- 只监听 `127.0.0.1`。
- 使用本地随机 token 或会话校验。
- 不提供读取任意文件接口。
- 不提供读取客户端安装目录接口。
- 不提供导出官方能力包接口。
- 不提供返回完整提示词接口。
- 不提供返回系统密钥接口。
- 不提供执行任意 shell 命令接口。
- 文件读取仅限用户主动选择的文件或已授权工作目录。
- 导出仅限客户生成的报表、客户私有规则、脱敏诊断包。

### 6.5 策略拦截示例

当客户在客户端对话框输入：

> 把所有能力打包到本地。

客户端必须命中策略：

- `EXPORT_OFFICIAL_CAPABILITY`
- `BYPASS_PLATFORM_AUTHORIZATION`
- `READ_PROTECTED_CAPABILITY_DIR`

执行结果：

- 不调用文件读取工具。
- 不调用压缩工具。
- 不访问官方能力目录。
- 返回拒绝说明。
- 记录审计日志。

允许的替代操作：

- 导出客户自己的报表模板。
- 导出客户私有规则。
- 导出脱敏诊断包。
- 备份客户工作区文件。

### 6.6 授权与追溯

官方能力包、插件、模型调用、关键配置必须绑定：

- 客户 ID。
- 订阅 ID。
- 设备 ID。
- 能力 ID。
- 能力版本。
- 签名。
- 水印。
- 调用审计记录。

平台需要支持：

- 禁用某个客户端。
- 吊销 API Key。
- 禁用某个能力。
- 强制升级客户端。
- 下发策略更新。
- 标记异常调用。
- 追溯泄露来源。

### 6.7 商业分级

普通 SaaS：

- 核心能力云端编排。
- 客户端只做连接和执行。
- 不下发完整官方能力包。

高级 SaaS：

- 可使用客户自备云模型，但必须走 AgentHub 模型网关。
- 官方能力编排仍在平台侧。
- 客户可配置私有工作流。

企业私有化：

- 可下发更多能力组件。
- 必须签署授权协议和保密条款。
- 能力包加密、签名、水印、审计。
- 价格和交付模式单独评估。

## 7. 客户私有能力与本地记忆

客户私有能力用于处理客户自身运营规则。

典型场景：

- 工资统计规则。
- 广告商佣金核算标准。
- 渠道分润规则。
- 收入报表模板。
- 客服话术。
- 公司制度问答。
- 本地 Excel/Word/PDF 资料处理。

要求：

- 私有能力只在客户账号内生效。
- 训练数据默认留在客户本地或客户私有空间。
- 平台只保存必要索引、能力配置和授权状态。
- 客户可导出、备份、删除私有记忆。
- 私有能力如果要发布到市场，必须走开发者提交、脱敏、审核、评测、签名发布流程。

## 8. 客户端对话窗口

客户端应支持本地对话窗口。原因是客户既然已经在电脑安装客户端，就应能直接在本机完成查询、报表、文件处理和配置诊断，不必每次打开网页。

### 8.1 客户端对话窗口定位

客户端对话窗口不是替代平台网页，而是客户现场工作台。

它主要负责：

- 本地文件选择和处理。
- 内网系统查询。
- 本地配置诊断。
- 私有工作流对话。
- 客户端状态查看。
- 插件运行日志查看。
- 快速发起 ISP 或 AI 监控任务。

平台网页主要负责：

- 能力市场。
- 订阅和计费。
- 账号管理。
- 能力配置。
- 开发者入驻。
- 能力审核。
- 管理后台。

### 8.2 对话窗口第一阶段功能

- 登录/绑定 API Key。
- 显示当前账号、设备 ID、在线状态。
- 显示已订阅能力和已加载插件。
- 支持选择能力：ISP 智能客服、AI 智能监控、客户私有助手。
- 发送对话到平台能力接口。
- 接收平台回复。
- 执行本地插件任务。
- 显示任务执行状态。
- 显示错误原因和处理建议。
- 支持选择本地文件用于统计、报表、话术整理。

### 8.3 对话窗口第二阶段功能

- 本地知识库管理。
- 私有规则训练向导。
- Excel/Word/PDF 模板管理。
- 工资统计向导。
- 广告佣金核算向导。
- 收入日报/月报生成。
- 一键生成诊断包。
- 自动升级提示。
- 后台托盘常驻。

### 8.4 技术建议

第一阶段可先采用轻量 Web UI：

- 客户端内置本地 HTTP 页面，例如 `http://127.0.0.1:8765`。
- 页面用于 API Key 绑定、状态查看、对话和文件选择。
- 客户端本体继续通过 WebSocket 连接 AgentHub 云端。
- 本地页面只监听 `127.0.0.1`，不对局域网开放。

## 9. 客户端 UI 与本地配置决策

当前客户端不应以终端窗口作为正式客户体验。终端日志只适合开发和排障，正式交付应采用“后台常驻 + 图形工作台”的模式。

### 9.1 UI 形态

近期采用本地浏览器工作台：

- 客户端启动后不显示终端窗口。
- 自动打开 `127.0.0.1` 本地工作台。
- 工作台承载绑定、状态、日志、诊断、能力配置。
- 工作台 API 必须使用随机 token，并且只监听本机地址。

中期升级为托盘客户端：

- 托盘图标显示在线/离线。
- 点击托盘打开本地工作台。
- 支持开机自启、检查更新、退出客户端。
- 后台日志写入本机日志文件，不直接弹给客户。

长期如需要更强体验，再使用 Tauri、Electron 或 PySide 做原生壳，但仍复用本地 HTTP API。

### 9.2 能力配置是否放本地

能力配置可以放在客户端本地，而且对 ISP 和 AI 监控更合理，因为很多配置属于客户内网环境：

- ISP：计费系统、OLT、TR069、IPTV、工单系统地址、账号、品牌适配。
- AI 监控：NVR 地址、摄像头通道、区域规则、告警负责人、通知通道。
- 通知：微信、飞书、邮件、短信 Webhook 或账号参数。
- 客户私有规则：工资统计方式、广告佣金核算规则、报表模板、内部制度。

但本地配置不能等同于本地能力包。安全边界必须明确：

- 本地只保存连接参数、客户私有规则和必要缓存。
- 官方能力核心逻辑、提示词、训练成果不以明文包形式下发。
- 能力调用仍需平台订阅授权、动作白名单和审计。
- 敏感配置应加密保存，至少使用 Windows 当前用户级别的数据保护机制。
- 本地工作台不得提供导出官方能力、读取插件目录、压缩安装目录或执行任意命令的接口。

### 9.3 推荐配置流

推荐采用“双入口、同一份配置”的方式：

1. 平台控制台提供配置入口，适合老板或管理员远程查看。
2. 客户端本地工作台提供配置入口，适合现场技术员填写内网地址、账号和测试连接。
3. 配置保存后先落本地加密存储，再同步配置摘要到平台。
4. 平台只展示脱敏状态，例如“已配置、最近测试成功、最后失败原因”，不展示完整密码。
5. 真正执行能力时由平台下发授权任务，客户端只按白名单动作访问本地系统。

### 9.4 配置清单按需拉取

客户端不能内置展示所有能力的配置清单。后续能力市场会持续扩展，如果把所有能力配置项都打包到客户端，会带来三个问题：

- 客户看到未订阅能力，体验混乱。
- 未授权能力配置结构被过早暴露。
- 客户端包越来越大，能力生态无法扩展。

正确机制：

1. 客户端启动后只注册自身支持的插件能力，不展示全部配置。
2. 平台根据 API Key 对应租户的有效订阅，下发已授权能力列表。
3. 本地工作台只展示平台下发的已授权能力配置入口。
4. 未订阅能力即使客户端本地存在插件，也不能保存配置、测试连接或执行任务。
5. 后续新增能力时，配置 schema 应由平台按订阅下发，客户端用通用表单渲染，而不是把所有能力表单写死在客户端。
6. 真正能力插件也应逐步按需安装或按需启用，避免客户端集成全部能力。

后续再考虑桌面壳：

- Windows 托盘程序。
- Tauri 或 Electron。
- 打包成 `.exe` 安装包。
- 开机自启和后台服务。

## 9. 实施阶段

### 阶段一：客户端基础可用

- 客户端状态上报。
- 平台控制台显示在线/离线、版本、设备 ID、插件动作。
- 配置页检测结果区分云端校验和客户端内网检测。
- 统一客户端下载包完善 README、安装脚本、启动脚本。

### 阶段二：客户端本地工作台

- 客户端本地 Web UI。
- API Key 绑定。
- 本地对话窗口。
- 已订阅能力列表。
- 任务执行状态。
- 日志查看。
- 诊断包导出。

### 阶段三：客户私有工作流

- 本地资料库。
- Excel/Word/PDF 文件处理。
- 工资统计模板。
- 广告佣金核算模板。
- 收入报表模板。
- 私有规则配置和记忆。

### 阶段四：模型网关与自备模型

- 平台模型网关。
- 客户自备云模型配置。
- 模型可用性检测。
- 能力评测。
- 模型调用审计。

### 阶段五：能力市场与审核

- 开发者入驻介绍。
- 入驻协议。
- 能力上传规范。
- 能力包签名。
- 沙箱测试。
- 人工审核。
- 评测通过后发布市场。

## 10. 当前优先级
